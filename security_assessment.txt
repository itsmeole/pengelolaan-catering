KONTROL KEAMANAN APLIKASI YANG PERLU DICATAT
================================================================================

Berikut adalah fitur keamanan yang diimplementasikan di level aplikasi dan belum
tercermin di item Zero Trust checklist standar, namun penting untuk dokumentasi:

A. ANTI-SPAM: BLOKIR PESANAN SAAT ADA ORDER AKTIF
   - Implementasi: /api/order/route.ts (POST - checkout siswa)
     dan /api/public/order/instan/route.ts (POST - instan order)
   - Logika: Sebelum memproses checkout baru, sistem SELECT dari tabel Order
     dengan filter studentId = user.id dan status IN ('PENDING', 'PAID').
   - Jika ada → tolak dengan HTTP 400 dan pesan eksplisit.
   - Efek: Satu siswa hanya bisa memiliki SATU pesanan aktif sekaligus.
   - Mencegah: double-order, manipulasi saldo, dan abuse sistem pesanan.

B. IP RATE LIMITING: INSTAN ORDER
   - Implementasi: src/lib/rateLimiter.ts (custom in-memory store)
   - Diaplikasikan di: /api/public/order/instan/route.ts (Layer 1)
   - Aturan: Max 1 request order per IP dalam window 3 menit.
   - Jika terlampaui → HTTP 429 dengan header Retry-After (dalam detik & menit).
   - IP diekstrak dari x-forwarded-for (Vercel proxy-aware) atau x-real-ip.
   - IP baru dicatat (recordRequest()) SETELAH validasi dasar lolos, bukan
     sebelumnya (mencegah IP exhaustion oleh request garbage).

C. SESSION-IDENTITY BINDING: CEGAH IMPERSONASI
   - Implementasi: /api/public/order/instan/route.ts (Layer 2)
   - Logika: Jika ada sesi login aktif, studentId dalam request body harus
     sama persis dengan user.id dari JWT session.
   - Jika berbeda → HTTP 403 Forbidden dengan pesan eksplisit:
     "Anda tidak dapat memesan atas nama orang lain saat sedang masuk (login)."
   - Efek: User yang sudah login tidak bisa memanfaatkan halaman /instan
     untuk memesan atas nama siswa lain.
   - Mode publik (tanpa login) tetap berjalan normal — bergantung pada
     kepercayaan operator fisik di kantin.

D. DATA SNAPSHOT INTEGRITY (IMMUTABLE TRANSACTION RECORD)
   - Implementasi: OrderItem menyimpan data snapshot saat transaksi:
     menuName, vendorName, vendorId, price, adminFee
   - Data ini tidak berubah meskipun menu/vendor data master diupdate.
   - Memastikan laporan historis akurat dan tidak bisa dimanipulasi
     dengan mengubah data master setelah transaksi.

E. REFUND LOCK: HANYA ADMIN YANG BISA REFUND
   - Siswa tidak memiliki endpoint atau UI untuk mengajukan refund sendiri.
   - Seluruh logika refund ada di /api/order/cancel yang hanya dapat
     dipanggil oleh Admin (diverifikasi via session role).
   - Mencegah: Penyalahgunaan sistem refund oleh siswa.

F. KONTROL INTEGRITAS MENU
   - Menu yang sudah pernah dipesan tidak bisa dihapus (cek FK constraint
     23502/23503 ditangkap dengan pesan error yang jelas ke vendor).
   - Mencegah hilangnya referensi data pada laporan historis.

G. SUPABASE AUTH BUILT-IN RATE LIMITS (PLATFORM LEVEL)
   - Aktif secara default tanpa konfigurasi tambahan:
     * /auth/v1/token    : 1800 req/jam per IP, burst 30
     * /auth/v1/verify   : 360 req/jam per IP, burst 30
     * /auth/v1/signup   : 60 detik window per user
     * /auth/v1/recover  : 60 detik window per user
     * /auth/v1/otp      : 360 OTP/jam
     * Anonymous sign-in : 30 req/jam per IP, burst 30
   - Melindungi dari brute force, credential stuffing, dan OTP abuse.

================================================================================
TEMUAN KRITIS (HIGH RISK) — YANG PERLU DIPERBAIKI
================================================================================

1. [KRITIS] Siswa Menggunakan NIS sebagai Password Default
   - NIS bersifat semi-publik (diketahui teman sekelas, guru, staf sekolah).
   - Jika NIS bocor, akun siswa dapat diakses oleh pihak yang tidak berwenang.
   - REKOMENDASI: Paksa siswa ganti password saat login pertama (forced change).
     Tambahkan kebijakan minimum password (min. 8 karakter, bukan hanya angka).

2. [KRITIS] Tidak Ada MFA untuk Akun Admin
   - Admin memiliki akses ke seluruh data keuangan, data pribadi, dan dapat
     mereset password semua pengguna. Kompromi akun admin = kompromi sistem penuh.
   - REKOMENDASI: Aktifkan TOTP MFA di Supabase Auth Dashboard untuk role ADMIN.

3. [SEDANG] IP Rate Limiter Instan Order Menggunakan In-Memory Store
   - Rate limiter saat ini menggunakan Map JavaScript yang tersimpan dalam
     memori process Node.js. Data hilang saat server restart atau cold start.
   - Di Vercel Serverless, setiap function instance punya memori sendiri,
     sehingga rate limit tidak efektif jika ada multiple instances.
   - REKOMENDASI: Migrasi ke persistent store (misalnya Upstash Redis via
     @upstash/ratelimit) untuk rate limiting yang benar di serverless.

4. [SEDANG] Halaman /instan Dapat Diakses dan Digunakan Tanpa Login
   - Siapapun yang mengetahui URL dapat memesan atas nama siapapun
     (kecuali jika sedang login — sudah ada binding check).
   - Bergantung pada kontrol fisik operator di kantin.
   - REKOMENDASI: Tambahkan PIN operator atau kode akses terbatas waktu
     untuk endpoint /instan.

================================================================================
REKOMENDASI PRIORITAS PERBAIKAN
================================================================================

PRIORITAS 1 - Segera (High Impact, Low Effort):
  a. Aktifkan MFA/TOTP untuk akun Admin di Supabase Dashboard (gratis, built-in).
  b. Tambahkan forced password change saat login pertama siswa.
  c. Migrasi IP rate limiter ke Upstash Redis untuk efektivitas di Vercel serverless.

PRIORITAS 2 - Jangka Pendek (1-3 bulan):
  a. Dokumentasikan klasifikasi data secara formal.
  b. Tambahkan audit log minimal untuk aksi kritis: login, refund, verifikasi bayar,
     perubahan data pengguna.
  c. Aktifkan Row Level Security (RLS) di Supabase sebagai defense-in-depth.
  d. Tambahkan proteksi akses /instan (PIN operator atau pembatasan origin).

PRIORITAS 3 - Jangka Menengah (3-6 bulan):
  a. Buat Incident Response Plan yang terdokumentasi (minimal: prosedur breach,
     eskalasi, notifikasi pengguna, rollback DB).
  b. Lakukan security briefing kepada Admin sekolah (phishing, password hygiene).
  c. Pertimbangkan enkripsi field-level untuk data PII sensitif (NIS, nomor HP).
  d. Integrasikan log monitoring terpusat (Sentry/Logtail) dengan alerting otomatis.
